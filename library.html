Project Name: Data Structure Library

Oberview

A library implementing common (and some uncommon but interesting) data structures in java. The idea here is largely to drill data structures in a way that will hopefully improve performance in technical interviews, in addition to testing with JUnit, which is a practice that I'm not going into this with any experience in.

This is also the first project where I am creating the write up in line with creating the program itself, so the rest of this will come in the form of regular dated updates.

List of Structures to be Implemented

  1. Linear
    1. Dynamic array
    2. Doubly linked list (optionally circular, optionally singly linked)
    3. Array based stack
    4. Linked list-based stack
    5. Array-based circular queue
    6. Deque
  2. Hashing
    1. Open addressing hash table
    2. Separate chaining hash table
  3. Trees
    1. Binary search tree
    2. AVL tree
    3. Red-black tree
    4. B-tree
    5. Min heap
    6. Max heap
    7. Fibonacci heap
    8. Standard trie (just for comprehension; will be replaced by the...)
    9. Compressed (radix) trie
  4. Graphs
    1. Adjacency list
    2. Adjacency matrix
    3. BFS and DFS graph traversal (probably methods)
    4. Disjoint set
  5. Advanced
    1. Skip list
    2. Bloom filter
    3. Fenwick tree
    4. Segment tree
    5. Suffix tree

1/31/25

Today I created the maven project and basic file structure for the Java library project, which I'm (somewhat regrettably) calling 'nickslibrary'. So there are now file stubs for all of the various data structures I plan to implement in the coming…weeks? Months? Days? The timeline is very much up in the air at the moment, but I'd like to knock out this project and the calculator by the end of the semester then maybe spend the summer developing something that isn't a homework assignment.

It's very likely that BFS and DFS will be implemented as methods in the graph files rather than as separate files.

2/2/25

I realized I forgot to add DynamicArray (main and test) in the original file structure, so I added those in. As by far the simplest data structure on the list, this is the first one I am creating. I started the skeleton of the file, basic methods that I know I'll use and a couple of constructors were added as stubs. Users can initialize one of my DynamicArrays empty, with a specification of an initial capacity, or by declaring the array with some variable quantity of objects T. The question of whether to use a generic T[] or an Object[] came up, and I went with Object[] to avoid excessive SuppressWarning(“unchecked”) flags.

2/5/25

Work has now been completed on the LinkedList, Array-Based Stack, and Linked List based stack. Both stack implementations were extremely easy, almost trivial, because the logic for accessing a dynamic array or linked list element at a given index (and inserting one at that index) already exists in the dynamic array and linked list classes themselves. Linked List was probably the most logically difficult thing to create, probably because that one class can, depending on flags, act as either singly or doubly linked and as either circular or non. I wound up finding a flow once it became clear that each add, search, or remove operation basically falls into the category of 'dealing with an empty list', 'addressing the item at the head', 'addressing the item at the tail', and 'other'. There are some small optimizations: if you have a doubly linked list and want to interact in some way with an element that is at index greater than size / 2, it's quicker to start at the tail and traverse to the element backwards. If you have a circular list, then it's easier to address any index as index % size to handle errors, rather than immediately declaring OOB something which really is not.

Testing, on the other hand, is another story. Half the point of this exercise is gaining an understanding of unit tests. Right now I have an issue where a certain test was failing, so I made changes to the code to resolve it. I think I see what the problem was, and I think my fix should work…but the test is still failing with the same error. I entered some print statements to get more info than I was getting from the test cases, and they are indicating that things are working well. I build the stack on top of the arraylist which was giving me the error and all THOSE test cases are passing. Somehow, I don't really understand why yet, my LinkedListTest.java file seems to be testing an old version of the linked list class. So we don't have these structures in the books quite yet, but they're getting there. By tomorrow this all should be figured out.

2/7/25

Today's big addition was the circular queue. I was struck by how simple this was to implement, the whole thing plus tests took like an hour. Which I think is decent time for all the tests to pass and for me to learn what a circular queue is.

It was quick enough that I went on to at least begin the deque implementation. At the moment, I am trying to decide between an array based implementation or a linked list based. This being an arbitrary library with no expected use-case, I might go with linked list just because of it's slightly higher conceptual complexity.

I wrote that while writing the skeleton of the file, figuring out my fields and constructors, and landed on linked list based because of the conceptual complexity being higher and there being no other use case. Then I spent 10 minutes writing a program that exclusively calls methods I already wrote in the linked list class, with no local logic at all, so I guess so much for conceptual complexity. Anyway, another data structure is in the books, because I did the real work a few days ago.
